# Blog All-in-One 1차 MVP

블로그 운영을 위한 콘텐츠 초안 생성 도구의 실행 가능한 MVP 뼈대입니다. 이번 단계는 실제 AI 완성형 기능보다 화면 구조, 선택형 생성 흐름, 로컬 임시 저장을 빠르게 확인하는 데 초점을 둡니다.

## 실행 방법

Windows PowerShell에서 `npm.ps1` 실행 정책 오류가 나면 `npm` 대신 `npm.cmd`를 사용합니다.

1. 의존성 설치

```bash
npm.cmd run install:all
```

2. 프론트엔드 실행

```bash
npm.cmd run dev:frontend
```

브라우저에서 `http://127.0.0.1:5173`으로 접속합니다.

3. 백엔드 실행

```bash
npm.cmd run dev:backend
```

백엔드 상태 확인 주소는 `http://localhost:4000/api/health`입니다.

## 크몽 시연용 데모 배포

정식 SaaS가 아니라 판매 전 시연용 URL이 목표라면 Render Web Service 하나에서 백엔드 API와 React 정적 파일을 함께 서빙하는 구조가 가장 단순합니다.

- 프론트엔드: `frontend/dist`로 빌드한 React 정적 파일
- 백엔드: Render Web Service의 Express 서버
- 저장 방식: 현재 MVP처럼 브라우저 `localStorage`
- 제외 범위: 회원가입, 결제, DB 저장, 실제 외부 발행 자동화

Express 서버는 `/api/...` 요청은 API 라우터로 처리하고, 그 외 `/`, `/comment-replies`, `/storage`, `/settings` 같은 프론트 라우트는 `frontend/dist/index.html`로 fallback 처리합니다. 그래서 배포 후 Render URL 하나로 앱 화면과 API health check를 함께 확인할 수 있습니다.

### 필요한 환경변수

Render Web Service에 입력합니다.

```text
CORS_ORIGIN=*
GEMINI_API_KEY=
PEXELS_API_KEY=
```

Render는 실행 포트로 `PORT` 환경변수를 사용할 수 있고, 별도 설정이 없으면 기본 웹 서비스 포트인 `10000`을 사용합니다. 로컬에서 직접 포트를 바꾸고 싶을 때만 `BACKEND_PORT=4000`을 사용합니다.

현재 1차 MVP는 Gemini/Pexels API를 실제 호출하지 않습니다. 나중에 연결할 때도 API 키는 프론트가 아니라 Render 백엔드 환경변수에만 넣어야 합니다.

### Render 단일 Web Service 배포

`render.yaml`을 사용하는 경우 현재 설정은 단일 Web Service 기준입니다.

수동으로 Render에서 설정한다면 아래처럼 입력합니다.

1. Render에서 New Web Service를 선택합니다.
2. GitHub 저장소 `mrsweet0519/a-blog-allinone`을 연결합니다.
3. Root Directory는 비워두거나 저장소 루트로 설정합니다.
4. Build Command는 아래 값을 사용합니다.

```bash
npm install --prefix frontend && npm install --prefix backend && npm run build --prefix frontend
```

5. Start Command는 아래 값을 사용합니다.

```bash
npm run start --prefix backend
```

6. Environment Variables에 필요한 값을 입력합니다.
7. Deploy 후 Render 서비스 화면의 URL로 접속합니다.

### 배포 후 접속 확인

예:

```text
https://a-blog-allinone.onrender.com
https://a-blog-allinone.onrender.com/comment-replies
https://a-blog-allinone.onrender.com/api/health
```

`/`에서 블로그 올인원 앱 화면이 뜨고, `/comment-replies`에서 댓글 응답 관리 화면이 뜨면 React 정적 파일 서빙이 정상입니다. `/api/health`에서 `{"ok":true}`가 보이면 백엔드 API도 정상입니다.

## 지인 테스트용 정적 베타 배포

지인 테스트처럼 빠르게 열리는 URL이 필요할 때는 Cloudflare Pages 또는 Vercel에 `frontend/dist`만 올리는 정적 베타 모드를 사용합니다. 이 모드는 기존 Render 배포와 로컬 개발 구조를 삭제하지 않고, 프론트 앱만 별도로 배포합니다.

정적 베타에서 사용할 수 있는 기능은 아래와 같습니다.

- 콘텐츠 메이커: 처음 사용 가이드, 예시 자동 채우기, 주제, 제목, 개요, 본문, 해시태그 그룹, 글 구성 체크
- 콘텐츠 메이커: 블로그 글 구조 점검, 키워드 자연 반영, 답변형 문장 구조, 본문 흐름 점검
- 콘텐츠 메이커: 본문 내 직관적인 이미지 삽입 위치 표시, 직접 촬영 추천/AI 이미지 프롬프트/이미지 사이트 검색어 카드, 본문만/본문+이미지 표시/이미지 검색어만 복사
- 판매용 접속 제어: 접속코드와 만료일 기준 간단 인증, 사용권 정보 표시, 코드 초기화
- 작성 프로필 다중 저장/불러오기와 초안 `localStorage` 저장
- 댓글 응답 관리: 작성자명 선택형 수동 댓글 입력, 캡처 이미지 Ctrl+V 붙여넣기, 캡처 이미지 업로드, 이미지 미리보기, OCR 원문/댓글 카드 분리, 수동 보정
- 추출 댓글 카드 반영, 개별/전체 대댓글 생성, 개별/전체 복사

정적 베타에서 비활성화되는 기능은 아래와 같습니다.

- URL에서 네이버 댓글 자동 불러오기
- 선택 댓글 자동 등록
- 전체 자동 등록
- 로컬 `blog-automation` 브리지 연결 (정식 상품화 시 고급 기능으로 분리됨)
- Express 백엔드 `/api/...` 의존 기능

앱은 Cloudflare Pages, Vercel, GitHub Pages 빌드 환경을 감지하면 화면 상단에 `정적 베타 모드`를 표시합니다. 이때 안내 문구는 다음처럼 표시됩니다.

```text
현재 베타 URL에서는 콘텐츠 생성, 캡처 업로드, 수동 댓글 응답 초안 생성을 테스트할 수 있습니다.
네이버 댓글 자동 수집/자동 등록은 로컬 blog-automation 브리지 연결 시 사용할 수 있습니다.
```

### 추천 플랫폼

지인 테스트용으로는 Cloudflare Pages를 먼저 추천합니다. 무료 정적 호스팅이 빠르고, 이 저장소에는 SPA 새로고침 404를 막기 위한 `frontend/public/_redirects` 파일이 포함되어 있습니다.

### Cloudflare Pages 설정

1. Cloudflare Dashboard에서 Workers & Pages로 이동합니다.
2. Create application 또는 Create project를 누릅니다.
3. Pages 탭에서 GitHub 저장소를 연결합니다.
4. 저장소 `mrsweet0519/a-blog-allinone`을 선택합니다.
5. Build settings에 아래 값을 넣습니다.

```text
Framework preset: None 또는 Vite
Build command: npm install --prefix frontend && npm run build --prefix frontend
Output directory: frontend/dist
```

6. Environment variables는 없어도 됩니다. 자동 감지가 안 되는 경우에만 아래 값을 추가합니다.

```text
VITE_DEPLOY_TARGET=static
```

7. Save and Deploy를 누릅니다.
8. 배포가 끝나면 Cloudflare Pages가 보여주는 `https://프로젝트명.pages.dev` 주소로 접속합니다.
9. 접속코드를 입력한 뒤 `/`, `/app`, `/comment-replies`, `/storage`, `/settings`가 새로고침해도 열리면 정상입니다.

### 판매용 접속코드 게이트

크몽 및 직접 판매를 고려해, URL만 아는 사람이 계속 접속하는 것을 막기 위한 1차 판매용 접속 제어 구조를 적용합니다. 정식 회원가입 전 단계로 `접속코드 + 만료일` 방식을 사용합니다.

- 코드 목록: `frontend/src/data/accessCodes.js`
- 검증 함수: `frontend/src/lib/accessControl.js`
- 진입 화면: `frontend/src/components/AccessGate.jsx`
- 인증 저장소: 브라우저 `localStorage` 인증 유지
- 앱 표시: 좌측 메뉴 하단에 사용권 이름, 만료일, active 여부, 코드 초기화 버튼 표시

검증 규칙:
- `code`가 목록에 존재해야 함
- `active` 상태가 `true`여야 함
- `expiresAt`(만료일)이 지나지 않아야 하며, 만료 시 즉각 접속 제한
- **예시**: `MGO-TEST7` (7일 테스트), `MGO-BETA30` (30일 사용권)

> **주의사항**: 현재 적용된 정적 베타용 프론트 기반 간단 인증은 완벽한 보안을 제공하지 않습니다. 정식 판매가 확대될 시에는 반드시 Supabase 또는 Cloudflare Workers/KV 기반의 서버 검증 방식으로 교체하는 것을 권장합니다.

### Vercel 설정

Vercel은 아래 방식 중 하나로 설정할 수 있습니다. 초보자에게는 Root Directory를 `frontend`로 지정하는 방식을 추천합니다.

```text
Root Directory: frontend
Framework Preset: Vite
Build Command: npm install && npm run build
Output Directory: dist
```

저장소 루트를 Root Directory로 유지한다면 아래처럼 설정합니다.

```text
Build Command: npm install --prefix frontend && npm run build --prefix frontend
Output Directory: frontend/dist
```

Vercel에서 직접 `/comment-replies`를 새로고침할 때 404가 나지 않도록 `frontend/vercel.json`에 SPA rewrite 설정을 넣어두었습니다. 배포 후 Vercel 프로젝트 화면의 Domains 또는 Deployment URL에서 테스트 주소를 확인하면 됩니다.

자동 감지가 안 되는 경우 Vercel Environment Variables에 아래 값을 추가합니다.

```text
VITE_DEPLOY_TARGET=static
```

### GitHub Pages 설정

GitHub Pages도 가능합니다. 다만 GitHub Pages는 정적 파일만 서빙하므로 backend/API/자동화 브리지는 작동하지 않습니다. 네이버 댓글 자동 수집/자동 등록은 로컬 `blog-automation` 브리지 또는 Render 백엔드 포함 배포에서만 테스트합니다.

GitHub Pages는 프로젝트 경로가 `/저장소명/`으로 붙기 때문에 base 경로와 라우터 설정을 맞춰야 합니다. GitHub Actions로 배포한다면 빌드 환경변수에 아래 값을 넣는 방식을 권장합니다.

```text
VITE_DEPLOY_TARGET=github-pages
VITE_ROUTER_MODE=hash
VITE_BASE_PATH=/a-blog-allinone/
```

이 설정을 쓰면 접속 URL은 보통 아래 형태가 됩니다.

```text
https://사용자명.github.io/a-blog-allinone/#/comment-replies
```

HashRouter를 사용하면 GitHub Pages에서 새로고침해도 404가 나지 않습니다.

### 정적 베타 접속 확인 경로

정적 베타 배포 후 아래 경로를 차례로 확인합니다.

```text
/
/app
/comment-replies
/storage
/settings
```

Cloudflare Pages와 Vercel은 SPA fallback 설정이 포함되어 있어 일반 경로로 열 수 있습니다. GitHub Pages는 HashRouter 설정을 사용할 때 `/#/comment-replies`처럼 `#` 뒤 경로로 접속합니다.

### 로컬 환경변수 예시

예시 파일은 아래에 있습니다.

- `frontend/.env.example`
- `backend/.env.example`

프론트에서 백엔드 API를 사용하려면 `frontend/.env`에 다음 값을 둡니다.

```text
VITE_API_BASE_URL=http://localhost:4000
```

백엔드 CORS 허용 주소를 바꾸려면 Render 또는 로컬 실행 환경에 다음처럼 넣습니다.

```text
CORS_ORIGIN=http://127.0.0.1:5173,http://localhost:5173,https://a-blog-allinone.onrender.com
```

Render 단일 Web Service 배포에서는 프론트와 백엔드가 같은 origin에서 동작하므로 `VITE_API_BASE_URL`을 따로 설정하지 않아도 프론트의 규칙 기반 fallback 기능은 동작합니다. 백엔드 API를 프론트에서 직접 호출하려면 같은 Render 주소를 `VITE_API_BASE_URL`로 넣어 빌드하면 됩니다.

## 폴더 구조

```text
a-blog-allinone
├─ frontend/            # Vite + React + Tailwind 화면
│  ├─ src/components/   # 공통 레이아웃, 상태 배지
│  ├─ src/lib/          # 로컬 저장, 목업 생성 로직
│  ├─ src/pages/        # 대시보드, 콘텐츠 메이커, 댓글 응답 관리, 보관함, 설정
│  └─ src/styles/       # 전역 Tailwind 스타일
├─ backend/             # Node.js + Express API 뼈대
│  └─ src/
│     ├─ api/           # API 라우터
│     └─ services/      # 콘텐츠 생성 서비스
├─ shared/              # 프론트/백엔드 공용 MVP 설정과 생성 로직
└─ docs/                # 기획/아키텍처/상세 명세 문서
```

## 1차 MVP 범위

- **기존 콘텐츠 메이커**: 키워드 기반 5대 주제별 포스팅 생성 (SEO/AEO/GEO 최적화 구조, 제목/개요/본문/해시태그 생성)
- **상품 후기 메이커**: 상품 이미지 + 사용자 메모 기반 자연스러운 후기형 포스팅 생성
- **개요 수정 및 본문 연동**: 제목 선택 후 생성된 개요 소제목을 직접 수정, 추가, 삭제할 수 있으며, Drag & Drop 및 모바일용 위/아래 이동 버튼을 통해 순서 변경 가능. 특히 **소제목별 메모(내용 힌트) 입력 기능**이 포함되어 수정한 개요와 메모가 실제 본문 생성에 반영됨.
- **주제별 결과 차별화**: 정보형/후기형/비교형/방문형/상품형의 5대 주제별로 본문 구조가 확실히 다르게 출력되며, 같은 키워드라도 선택한 주제에 따라 첫 문단, 소제목, 마무리(결론) 구조가 차별화됨.
- **친근한 구어체 말투 강화**: 친근한 말투는 '~했어요, ~더라고요, ~느껴졌어요, ~같아요' 어미를 중심으로 작성되며 딱딱한 보고서체 표현을 배제함. (콘텐츠 메이커 및 상품 후기 메이커 공통 적용)
- **소제목 출력 개선**: 결과 화면에서는 소제목이 굵게 보이도록 CSS 처리하며, 복사 및 네이버 블로그 스마트에디터 붙여넣기 시 `*`, `**`, `###` 같은 마크다운 특수문자 없이 깔끔한 순수 텍스트와 줄바꿈(개행) 형태를 유지.
- **SEO/AEO/GEO형 14종 원고 패키지**: 메인 키워드, 보조 키워드 5개, 검색 의도 분석, 홈피드 클릭 포인트, 제목 후보 5개, 최종 추천 제목 1개, 첫 문장 후보 3개, 블로그 본문, 사진 배치 가이드, 업체 정보 또는 상품 정보 정리, 이런 분께 추천해요, FAQ 3개, 해시태그 10~15개, 최종 검수표로 구성된 통합 원고 패키지 출력.
- **카테고리별 필수 확인 기준**: 맛집, 카페, 아이와 갈만한 곳, 상품 리뷰, 생활용품/계절템, 체험단/협찬 리뷰별 필수 확인 항목을 검증하며, 제공되지 않은 정보는 임의 지어내지 않고 `[확인 필요]`로 표시. 협찬 여부 미입력 시 `[협찬 여부 확인 필요]`로 최종 검수표에 강제 표시.
- 주제 추천 3개
- 목적별 제목 후보 5개: 정보형, 후기형, 비교형, 선택형, 클릭형
- 선택한 제목 기준 개요 소제목 3~6개에 대한 직접 수정, 추가, 삭제, 순서 변경(Drag & Drop 및 위/아래 버튼) 및 개별 메모 입력 지원
- 입력 도움말 툴팁(초보자용 키워드/예시 가이드 포함), 자동 예시 채우기, 기본/고급 입력 분리
- 처음 사용 가이드: PC 카드형, 모바일 접힘형
- 같은 입력값에서도 `다시 생성` 시 새로운 관점의 주제 후보 3개 생성
- 작성 프로필(브랜드/상품별) 다중 저장/불러오기/초기화 지원, 브라우저 `localStorage` 기준
- 첫 문장 후보 3개 선택
- CTA 문장 후보 3개 선택
- 선택한 주제/제목 기준 게시용 최종본 1개
- 게시용 최종본과 내부 전략 메모 분리
- 블로그 글 구조 점검: 제목 키워드, 첫 문장 키워드, 첫 문단 키워드 반복, 보조 키워드 분산, 제목/본문 일치, 검색 의도/목적 일치, 경험/체크포인트 포함, 금칙어/과장 표현 여부 확인

- 목표 글자수 선택: 1200자, 1500자, 2000자, 2500자, 직접입력
- 목표 글자수 ±10% 범위 안내
- 내부 관리자용 키워드 최적화 규칙
- 글 목적별 생성 구조: 정보 전달형, 신뢰 형성형, 상품 홍보형, 방문 유도형
- 사용자 유형별 결과 분기: 사업자/매장 홍보, 인플루언서/수익형
- 광고주형 입력값: 브랜드명/매장명, 지역, 핵심 강점, 강조 포인트, CTA 방향, 주소, 영업시간, 가격/가격대, 구매처, 문의 방법, 배송/교환 정보

- 검색패턴 기반 해시태그 그룹 생성 및 네이버 블로그에 바로 붙여넣을 수 있는 전체/그룹별 복사 지원

- 접속코드와 만료일 기반 1차 판매용 게이트, 사용권 정보 표시, 코드 초기화
- 입력 전, 입력 완료, 생성 중, 생성 완료, 수정 중, 저장됨, 복사 완료 상태 표시

## 콘텐츠 메이커 흐름

1. 처음 사용하는 경우 상단의 `처음 사용 가이드`를 확인합니다. PC에서는 카드형으로, 모바일에서는 접힘형으로 표시됩니다.
2. 키워드가 막히면 매장 홍보, 상품 후기, 정보글 예시 중 하나를 골라 `예시 자동 채우기`를 사용합니다.
3. `기본 입력`에서 키워드, 업종/주제, 사용자 유형, 글 목적, 목표 글자수, 말투를 입력합니다.
4. 브랜드명/매장명, 지역, 핵심 강점, 강조 포인트, CTA 방향, 주소, 영업시간, 가격/가격대, 구매처, 문의 방법, 배송/교환 정보, 금지어는 `고급 입력 열기`를 눌러 필요할 때만 입력합니다.
5. 입력칸 설명은 라벨 옆 물음표 툴팁에서 확인합니다. PC에서는 마우스를 올리고, 모바일에서는 눌러서 볼 수 있습니다.
6. 반복해서 쓰는 품목이나 브랜드가 있다면 상단의 `작성 프로필 관리`에서 프로필 이름을 입력하고 `프로필 저장`을 누릅니다.
7. 저장된 작성 프로필은 목록에서 선택한 뒤 `불러오기`를 누르면 현재 입력값에 반영됩니다.
8. `주제 생성`을 눌러 추천 주제 3개를 만듭니다.
9. 마음에 드는 흐름이 아니면 `다시 생성`을 눌러 이전 후보와 겹치지 않는 새 관점의 주제 3개를 받습니다.
10. 추천 주제 중 1개를 선택합니다.
11. `제목 생성`을 눌러 정보형, 후기형, 비교형, 선택형, 클릭형 제목 후보를 만듭니다.
12. 제목 후보 중 1개를 선택합니다.
13. `개요 생성`을 눌러 소제목을 만든 뒤, 화면에서 소제목의 내용을 직접 수정하거나 `소제목 추가`, `삭제` 버튼으로 소제목을 편집합니다.
14. 모바일 및 PC 대응을 위해 **Drag & Drop** 또는 **위/아래 이동 버튼(▲/▼)**을 사용하여 소제목의 순서를 자유롭게 조정합니다.
15. **소제목별 메모 입력란**에 각 소제목 문단에 들어갈 구체적인 힌트나 경험 메모를 입력할 수 있습니다.
16. 첫 문장 후보(3개)와 CTA 후보(3개) 중 마음에 드는 문장을 선택합니다.
17. `본문 생성`을 누르면 수정된 개요 순서와 소제목별 메모가 AI 프롬프트에 실시간 반영되어 본문 작성이 진행됩니다.
18. 오른쪽 상단 결과 영역에 **SEO/AEO/GEO형 14종 원고 패키지**(메인/보조 키워드, 검색 의도 분석, 홈피드 클릭 포인트, 제목 후보 5개, 최종 추천 제목 1개, 첫 문장 후보 3개, 블로그 본문, 사진 배치 가이드, 업체/상품 정보 정리, 이런 분께 추천해요, FAQ 3개, 해시태그 10~15개, 최종 검수표)가 렌더링됩니다.
19. 블로그 글 구조 점검(검색에 잘 맞는 글인지 확인)에서 제목, 첫 문단, 소제목, 키워드, 마무리 문장 및 카테고리별 필수 항목 기입 여부가 정상인지 최종 검수표와 함께 확인합니다.
20. 게시용 본문 안의 `[여기에 이미지를 넣어주세요 이미지 N]` 표시와 하단 이미지 추천 카드의 번호를 맞춰 직관적으로 이미지 삽입 위치를 확인합니다.
21. 본문 복사 시, 소제목의 마크다운 특수문자(`*`, `**`, `###`)는 제거되고 굵은 글씨 효과는 제외되지만 네이버 에디터에 붙여넣기 가장 좋게 빈 줄(개행)이 정렬된 형태로 복사됩니다.

## 상품 후기 메이커 흐름

상품 후기 메이커는 크몽 및 직접판매 시 "상품 상세 이미지 여러 장과 간단한 메모만 넣으면 후기형 블로그 글 초안을 만든다"는 방향으로 판매 가능성을 높이는 강력한 기능입니다.

1. `상품명/브랜드명`과 `메인 키워드`를 입력합니다. 첫 화면은 이처럼 필수 정보와 이미지 업로드, 경험 메모 중심으로 간결하게 구성됩니다.
2. `상품 상세 이미지`를 여러 장 등록합니다. 파일 업로드와 `Ctrl+V` 캡처 붙여넣기를 모두 지원(여러 번 붙여넣으면 누적)하며, 모바일에서도 부담 없게 이미지 미리보기/삭제/정렬을 제공합니다.
3. **사진에서 읽은 내용 (추출/보정 영역)**: 이미지가 없는 초기 상태에서는 해당 영역이 접혀 있으며, "상품 이미지를 넣으면 읽은 내용을 확인할 수 있습니다."라는 안내만 표시됩니다.
4. 이미지를 넣으면 추출 정보 확인 영역이 자동 펼쳐지며, 이미지에서 추출된 상품 정보(상품명, 브랜드명, 성분, 가격, 특징 등)를 확인 및 보정할 수 있습니다. (※ OCR 추출 원문 전체는 보조 접힘 영역으로 분리하여 긴 스크롤 방지)
5. `내가 느낀 점 또는 쓰고 싶은 내용`을 간단히 메모합니다(경험 메모 입력).
6. 모바일에서도 생성 버튼까지 도달하는 흐름을 짧게 유지하기 위해 긴 정보는 기본 접힘으로 관리합니다. `강조 포인트`와 `금지어`, `말투`를 확인 후 생성 버튼을 누릅니다.
   - 결과물 구조: 도입부 → 제품을 찾게 된 이유 → 상황 공감 → 제품을 보게 된 이유 → 사용 방법 또는 루틴 → 느낀 점 또는 기대 포인트 → 이런 분께 맞을 것 같아요 → 구매 전 확인할 점 → 제품 정보 정리
6. 결과물을 `복사`하여 활용합니다.
7. **주의사항 자동 적용**: 광고문이 아닌 후기형 말투로 작성되며, 실제 사용 경험을 단정하지 않습니다. 건강/다이어트/장 관리/클렌즈 관련 과장 표현이나 효능 보장 표현을 엄격히 금지하고 개인차 안내를 자연스럽게 포함합니다. 특히 게시용 본문에는 "경험 메모:", "OCR 원문", "추출 데이터", "개발자용 구조명" 등 내부 라벨이 절대 노출되지 않습니다.

## 댓글 응답 관리 흐름

`댓글 응답 관리`는 블로그 운영자가 원댓글을 검토한 뒤 자연스러운 대댓글 초안을 복사해서 등록할 수 있게 돕는 보조 기능입니다. 1차 MVP는 자동등록이 아니라 `검토 후 등록`을 기준으로 합니다.
핵심 흐름은 "댓글 입력 → 상호대댓글 생성 → 복사"입니다.

1. `블로그 포스팅 URL`과 `포스팅 제목`을 입력합니다.
2. `메인 키워드`를 비워두면 포스팅 제목에서 후보를 자동 추출합니다. (키워드 예시는 실제 업체명이 아닌 중립적이고 검색 의도가 분명한 예시 사용)
3. 브랜드명/매장명, 지역, 사용자 유형, 말투, 금지어, CTA 톤, 내 블로그 닉네임, owner aliases를 필요에 맞게 입력합니다. (필수 입력값 최소화, 포스팅 제목 필수 표시 제거, 작성자명은 선택값으로 변경)
4. 입력 방식 선택(수동, 캡처 업로드 등)은 상단 탭/버튼으로 고정되어 헷갈리지 않게 제공합니다.
5. 댓글은 `수동 댓글 입력` 탭에서 붙여넣거나 `댓글 추가`로 카드형 입력을 추가합니다. 작성자명을 넣지 않고 댓글 내용만 넣어도 바로 대댓글을 생성할 수 있습니다.
6. `캡처 이미지 업로드` 탭(모바일 환경 대응 완비)에서는 네이버 블로그 댓글 화면 캡처를 파일로 올리거나, 파일 저장 없이 붙여넣기 영역을 클릭한 뒤 `Ctrl+V`로 바로 붙여넣을 수 있습니다.
7. 이미지 미리보기와 OCR 추출 텍스트를 보며 내용을 보정합니다. (댓글 추출과 댓글 추가의 역할을 명확히 분리) OCR이 실패하면 샘플 댓글을 넣지 않고 직접 수정 안내만 표시합니다.
8. `추출 결과 반영`을 누르면 작성자와 댓글 내용이 댓글 카드 목록으로 들어가며, OCR이 틀린 부분은 카드에서 다시 수정할 수 있습니다.
9. `전체 생성` 또는 행별 `생성`을 누르면 댓글 유형, 감정/의도, 핵심 키워드, 대댓글 초안, 메인 키워드 반영 여부, 금지어 포함 여부, 중복 위험도를 함께 표시합니다. 다시 생성은 기존 문장과 다른 variation을 제공합니다.
10. 결과는 개별 대댓글 복사, 전체 대댓글 목록 복사, 원댓글 + 대댓글 세트 복사를 지원합니다.
11. 검토가 끝난 행은 `검토` 버튼으로 표시하고, 답변하지 않을 댓글은 `스킵`으로 구분합니다.
12. 현재 작업은 브라우저 `localStorage`에 임시 저장, 불러오기, 초기화할 수 있습니다.

### 댓글 캡처 이미지 붙여넣기

댓글 캡처 이미지는 파일로 저장하지 않고 바로 붙여넣을 수 있습니다.

1. 네이버 블로그 댓글 영역을 `Win + Shift + S`로 캡처합니다.
2. 댓글 응답 관리의 `캡처 이미지 업로드` 탭을 엽니다.
3. 붙여넣기 영역을 클릭합니다.
4. `Ctrl+V`로 캡처 이미지를 붙여넣습니다.
5. 이미지 미리보기와 추출 결과를 확인합니다.
6. 필요하면 추출 텍스트나 댓글 카드를 수정합니다.
7. `전체 생성` 또는 댓글별 `생성`으로 대댓글 초안을 만듭니다.

붙여넣기 영역은 `image/png`, `image/jpeg`, `image/webp` 형식을 지원합니다. 기존처럼 이미지 파일 선택 업로드도 계속 사용할 수 있으며 모바일 환경에서도 정상 작동해야 합니다.

OCR은 캡처 품질, 글자 크기, 배경색에 따라 틀릴 수 있습니다. 그래서 앱은 이미지에서 읽은 원문 텍스트와 자동 분리된 댓글 카드를 먼저 보여주고, 사용자가 작성자명과 댓글 내용을 수정한 뒤 `댓글 카드로 반영`하도록 구성되어 있습니다. 정적 베타 URL에서는 자동등록이 아니라 대댓글 초안 생성과 복사 테스트에 집중합니다.

### 콘텐츠 메이커 블로그 글 구조 점검 기준

콘텐츠 메이커는 글 생성 후 아래 항목을 점검합니다. 표현은 블로그 운영 보조 시스템에 맞춰 구성하며, 검색 결과 보장처럼 보이는 문구는 사용하지 않습니다.

- 메인 키워드가 제목에 포함됨
- 첫 문장에 메인 키워드가 자연스럽게 반영됨
- 첫 문단에 메인 키워드가 2~3회 자연스럽게 반영됨
- 첫 문단이 정보형 답변으로 시작함
- 보조 키워드가 본문과 소제목에 자연스럽게 분산됨
- 제목과 본문 주제가 일치함
- 검색 의도와 글 목적이 일치함
- 소제목이 본문 내용과 연결됨
- 경험/후기/비교/체크포인트 중 하나 이상 포함됨
- 동일 키워드 과다 반복 없음
- 사용자 유형과 글 목적이 본문 흐름에 반영됨
- 목표 글자수에 맞는 소제목 수가 적용됨
- 본문 안 어느 문단 뒤에 이미지를 넣을지 직관적으로 표시됨 (`[여기에 이미지를 넣어주세요 이미지 N]`)
- FAQ 또는 질문형 답변 구조가 포함됨
- 과장 표현 없음
- 마무리 CTA가 자연스럽게 들어감
- 금지어 없음

본문은 첫 문단에서 검색자가 궁금해할 답을 먼저 제시하고, 각 소제목의 첫 문장을 짧은 결론형 문장으로 시작합니다. 정보 전달, 상품 홍보, 비교형 제목처럼 자연스러운 경우에는 글 하단에 FAQ 2~3개를 추가합니다. 사업자/매장 홍보 또는 쇼핑몰/상품형 글에서 입력값이 있는 경우에는 본문 하단에 제품/매장 정보 정리를 자연스럽게 덧붙입니다.

### 댓글 유형 분류 기준

- `칭찬형`: 감사와 댓글 핵심 반영
- `공감/제품고민공감형`: 공감과 짧은 자연 반응, 사용자의 고민 포인트 반영
- `경험공유형`: 경험 인정과 감사
- `질문형`: 답변 우선, 짧은 감사
- `방문/구매의사형`: 반가움과 부드러운 구매/안내 유도
- `기대감표현형`: 써보고 싶다, 기대된다 같은 긍정 기대 표현 반영
- `사용감반응형`: 제품/서비스의 구체적 사용감에 맞춘 자연스러운 답변
- `짧은반응형`: 짧고 자연스러운 답변
- `불만/민감형`: 방어적 표현 없이 중립 응답
- `광고/스팸형`: 대댓글 생성 없이 스킵 권장
- `의미불명형`: 짧은 감사 또는 검토 필요

### 대댓글 생성 규칙

대댓글은 1~2문장으로 제한하고, 원댓글 내용에 직접 반응하는 말투를 우선합니다. 형식적인 템플릿 답변을 완전히 탈피하여 댓글의 감정, 의도, 구매 의사 등을 세밀히 반영합니다. 하얀가루, 가루날림, 백탁, 떡짐, 귀차니즘, 편리함, 구매 의사처럼 원댓글에 나온 핵심 표현을 먼저 잡아 답변에 반영합니다. **질문형이 아닌 댓글에는 "문의하신 부분은...", "방문 목적에 맞춰...", "먼저 확인해보시면 좋습니다" 같은 기계적 멘트를 절대 사용하지 않습니다.** 물음표만으로 질문형으로 오판하지 않고 실제 질문 의도가 있을 때만 질문형으로 처리합니다. 같은 문장이나 같은 감사 표현이 반복되지 않도록 최근 생성 답변과 비교하며, `다시 생성` 시 이전 문장과 명확한 variation이 도출되어야 합니다. 중복 위험도는 `중복 위험 낮음`, `중복 주의`, `재생성 권장`으로 표시합니다.

금지어에는 기본적으로 `최고`, `무조건`, `보장`을 포함하고, 사용자가 입력한 금지어를 추가로 검사합니다. 질문 댓글은 질문에 먼저 답하고, 짧은 댓글은 짧게 답하며, 광고/스팸 댓글과 내 계정 댓글, 이미 내 계정 대댓글이 있는 댓글은 스킵 구조로 처리합니다.

### SEO/메인 키워드 반영 원칙

댓글 응답 관리의 SEO는 키워드 반복이 아니라 포스팅 주제와 댓글 맥락을 자연스럽게 연결하는 방식입니다. 메인 키워드는 댓글 내용과 어울릴 때만 0~1회 반영하며, 모든 대댓글에 넣지 않습니다. 댓글과 맞지 않으면 키워드를 넣지 않는 것이 기본입니다.

### 향후 자동화 브리지

현재 URL 기반 댓글 수집과 자동등록은 1차 MVP 범위에서 제외합니다. 대신 `shared/commentReplyGenerator.js`와 백엔드 `/api/comment-replies` 라우터에 URL 수집 브리지 슬롯을 준비해두었고, 이후 기존 `blog-automation` 기능과 연결할 수 있는 구조로 분리해두었습니다.

### 말투 선택 기준 명확화 (콘텐츠/상품 후기 공통)

메이커에서 선택하는 말투는 다음과 같은 기준으로 일관되게 작동합니다:
- **친근한**: 자연스러운 블로그 후기 말투, ~해요/~더라고요/~같아요 중심
- **차분한**: 부드러운 설명형
- **전문적인**: 객관적 정보형
- **활기찬**: 밝고 경쾌한 후기형

## 초보자 기준 사용 흐름

처음 쓰는 사용자는 모든 칸을 채우려고 하지 않아도 됩니다. 기본 입력만 먼저 채우고, 모르는 항목은 고급 입력을 닫아둔 채 주제 후보를 확인하는 흐름이 가장 쉽습니다. 앱 내에서 제공되는 가이드와 '예시 자동 채우기' 기능에는 특정 실존 브랜드가 아닌 누구나 이해하기 쉬운 범용 상품군 예시가 적용됩니다:
- 수분크림 / 피부 보습 / 데일리 크림
- 무선 미니청소기 / 원룸 청소 / 생활용품
- 아기 식판 / 이유식 준비 / 육아용품
- 단백질 쉐이크 / 아침 대용 / 건강 루틴
- 여름 샌들 / 편한 신발 / 데일리룩

- 필수 입력: 키워드, 업종/주제, 글 목적, 말투
- 공란 가능 안내: 각 입력칸의 물음표 툴팁에서 확인
- 반복 입력 줄이기: 품목이나 브랜드별 작성 프로필을 저장한 뒤 필요할 때 목록에서 선택해 불러오기
- 흐름 조정: 자동 개요를 그대로 쓰지 않고 광고주가 원하는 순서로 소제목을 수정, 추가, 삭제
- 새 관점 찾기: 같은 입력값이어도 `다시 생성`을 눌러 이전 주제 후보를 피한 다른 표현과 관점 확인

## 입력 도움말과 화면 구조

콘텐츠 메이커의 입력 영역은 `기본 입력`과 `고급 입력`으로 나뉩니다. 기본 입력에는 글 생성에 바로 필요한 핵심 항목을 두고, 브랜드명/매장명, 지역, 강점, 강조 포인트, 금지어, CTA 톤, 이모지 사용 여부는 접힌 고급 입력 안에 둡니다.

긴 설명문은 화면에 계속 노출하지 않고 라벨 옆 물음표 툴팁으로 제공합니다. 툴팁은 제목, 짧은 설명, 예시를 나누어 보여주며, PC에서는 마우스를 올리고 모바일/터치 환경에서는 물음표를 눌러 확인합니다. `공란 가능` 안내와 예시도 이 툴팁 안에서 짧게 확인합니다.

## 개요와 주제 재생성 방식

개요는 제목 선택 후 자동 생성되지만 최종 본문에 들어가기 전 사용자가 직접 손볼 수 있습니다. 화면에서는 소제목 입력칸, `소제목 추가`, `삭제` 버튼을 제공하며, 선택된 소제목 3~6개일 때 본문을 만들 수 있습니다. 저장한 초안에는 사용자가 수정한 개요가 함께 보관됩니다.

주제 재생성은 단순히 같은 후보를 다시 출력하지 않습니다. 프론트에서 `regenerationCount`, `variationSeed`, 이전에 생성된 주제 목록을 함께 전달하고, 공용 생성 로직은 이미 나온 주제를 먼저 제외한 뒤 다른 관점의 후보 3개를 고릅니다.

## 작성 프로필 (브랜드/상품 프로필)

콘텐츠 메이커의 `작성 프로필 관리`는 다양한 품목을 운영하는 사용자를 위해 자주 쓰는 업종, 말투, 금지어, 글 목적, 키워드 스타일을 브라우저 `localStorage`에 다중 저장해둘 수 있는 기능입니다. 수분크림 후기 프로필, 무선 미니청소기 프로필, 피부관리샵 프로필, 초등 독서노트 리뷰 프로필처럼 여러 개를 저장하고 불러올 수 있습니다.

저장 대상은 사용자 유형, 업종/주제, 글 목적, 말투, 이모지 사용 여부, 금지어, CTA 톤, 목표 글자수, 브랜드명/매장명, 지역, 핵심 강점, 강조 포인트입니다.

- `프로필 이름`: 저장할 프로필 이름을 입력합니다.
- `프로필 저장`: 현재 입력값을 새 작성 프로필로 저장합니다.
- `저장된 작성 프로필`: 저장된 프로필 목록에서 불러올 항목을 선택합니다.
- `불러오기`: 선택한 작성 프로필을 현재 입력 화면에 반영합니다.
- `삭제`: 선택한 작성 프로필을 삭제합니다.

## 게시용 최종본과 내부 메모

메이커 생성 결과는 2단으로 나뉩니다.

- `게시용 최종본`: 광고주가 네이버 블로그에 바로 옮겨 쓸 수 있는 본문입니다. 내부 전략 설명, 메타 문장, 작성자용 안내 문구는 물론 "경험 메모:", "OCR 원문", "추출 데이터", "개발자용 구조명" 등 일체의 내부 라벨이 완전히 제거됩니다.
- `운영 메모 / 내부 전략 데이터`: 접힌 영역에 보관되는 보조 정보입니다. 펼치면 선택값 요약, 본문 흐름 점검, 이미지 사이트 검색어와 추천 컨셉만 간결하게 확인할 수 있습니다.

`복사` 버튼은 게시용 제목, 본문, 해시태그만 복사합니다.

## 이번 품질 튜닝 및 베타 피드백 반영 사항 (1차 베타)

1차 베타 테스트 피드백을 기반으로 콘텐츠 메이커와 댓글 응답 관리의 사용성 및 품질을 대폭 개선했습니다.

1. **콘텐츠 메이커 온보딩 강화**: 초보자가 키워드 입력을 쉽게 이해할 수 있도록 PC 카드형, 모바일 접힘형 '처음 사용 가이드'를 추가하고, 매장 홍보/상품 후기/정보글 등 상황별 '예시 자동 채우기' 기능을 제공합니다.
2. **작성 프로필(브랜드/상품 프로필) 다중 저장**: 품목이나 브랜드별로 작성 프로필을 여러 개 저장하고, 목록에서 선택해 현재 입력값에 바로 불러올 수 있게 개선했습니다.
3. **직관적인 이미지 삽입 위치 가이드**: 본문 안에 `[여기에 이미지를 넣어주세요 이미지 N]` 문구를 표시해 사용자가 어느 문단 뒤에 이미지를 넣어야 하는지 직관적으로 이해할 수 있게 개선했습니다.
4. **이미지 추천 구조 개선**: 단순 사이트 검색어 제공을 넘어 실무 중심으로 카드 구성을 개편했습니다. (이미지 컨셉, 직접 촬영 추천(블로그용 우선), AI 이미지 프롬프트(티스토리/SNS용), 이미지 사이트 검색어, 복사 버튼 제공)
5. **해시태그 복사 개선**: 네이버 블로그에 바로 붙여넣을 수 있도록 전체 해시태그 복사와 그룹별 해시태그 복사 기능을 강화했습니다.
6. **블로그 글 구조 점검 문구 개선**: "검색 의도 기반 구조 점검" 대신 사용자 친화적인 "블로그 글 구조 점검", "검색에 잘 맞는 글인지 확인" 문구를 사용하여 고객이 이해하기 쉽게 다듬었습니다.
7. **상호대댓글 품질 고도화**: 단순 템플릿 답변이 아닌, 원댓글의 핵심 표현(하얀가루, 떡짐, 귀차니즘 등)과 감정/의도/구매 의사를 먼저 파악해 자연스럽게 반영하도록 개선했습니다.
8. **댓글 유형 정밀화 및 오판 방지**: 제품고민공감형, 사용감반응형, 구매의사형, 기대감표현형, 경험공유형 등으로 유형을 세분화했습니다. 단순 물음표만으로 질문형으로 오판하지 않으며, "문의하신 부분은", "방문 목적에 맞춰" 등 부자연스러운 기계적 템플릿 문구를 금지어 목록으로 엄격히 차단했습니다.
9. **수동 댓글 입력 UX 간소화**: 포스팅 제목 필수 표시를 제거하고, 작성자명을 선택값으로 변경했습니다. 작성자명이 없어도(자동으로 댓글 1로 표시) 댓글 내용만으로 대댓글 생성이 가능합니다.
10. **다시 생성(Variation) 기능 개선**: 대댓글 다시 생성 시 기존 생성 문장과 중복되지 않는 새로운 Variation을 제공하도록 알고리즘을 보강했습니다.
11. **기능 모듈화 및 배포 환경 최적화**:
   - **정적 베타 모드**: URL 자동 수집/등록 기능이 없어도 콘텐츠 메이커, 캡처 업로드, 수동 댓글 입력 기반 대댓글 생성/복사 기능이 완벽히 동작합니다.
   - **고급 기능 분리**: URL 기반 자동 수집 및 등록은 로컬 `blog-automation` 브리지 기반의 정식 고급 기능으로 분리해 별도 관리합니다.

## 네이버 블로그 글쓰기 흐름에 맞춘 보조 시스템 구조

이 MVP는 단순 자동 글 생성기가 아니라 검색 의도 기반 글 구성을 돕는 **'블로그 운영 보조 시스템'**입니다. 상위노출 보장, 최적화 보장 등의 표현은 배제하고, 키워드의 자연스러운 반영을 도우며 철저히 **네이버 블로그 글쓰기 흐름에 맞춘 구조**와 **답변형 문장 구조**를 제안합니다.

### 1. 제목 기준
- **길이**: 30자 내외
- **키워드**: 메인 키워드 필수 포함
- **구조**: 정보성 + 경험 기반 내용 조합
- **유도**: 독자가 클릭할 분명한 이유가 있는 제목
- **제한**: 금칙어 사용 금지
- **맞춤화**: 사용자 유형과 글 목적에 따라 제목 톤 분리 (정보형, 후기형, 비교형, 선택형, 클릭형)

### 2. 첫 문단 기준
- **시작**: 첫 문장에 메인 키워드 사용
- **밀도**: 첫 문단 내에 메인 키워드 또는 자연 변형 키워드를 2~3회 반영
- **흐름**: 단순 인사말이 아닌 정보형 문장으로 시작
- **톤앤매너**: 과도한 광고 느낌 금지

### 3. 본문 구조
- **답변형**: 첫 문단에서 검색자가 가장 궁금해할 답변을 먼저 제시
- **소제목**: 각 소제목의 시작은 결론형 문장으로 구성
- **전개**: 정보형 내용에 경험 기반 흐름을 더하여 자연스러운 문맥 구성
- **FAQ**: 목적에 따라 FAQ 또는 질문형 구조를 선택적으로 적용
- **마무리**: 본문 하단에 업체/상품 정보를 자연스럽게 정리

### 4. 해시태그 기준
- **개수**: 10~15개 생성
- **구성**: 메인 키워드, 보조 키워드, 지역명 + 카테고리, 카테고리 + 후기, 브랜드/상품명, 검색 의도형 롱테일 포함
- **기능**: 네이버 블로그에 바로 붙여넣기 편하도록 전체 복사 및 그룹별 복사 지원

### 5. 블로그 글 구조 점검 (검색에 잘 맞는 글인지 확인)
게시용 최종본 아래에는 어려운 '검색 의도 기반 구조 점검' 대신 고객 친화적 명칭인 `블로그 글 구조 점검` 패널이 표시됩니다. 이 패널은 "작성된 글이 검색에 잘 맞는지" 직관적으로 확인하도록 돕습니다.

- 제목 키워드 반영 여부
- 첫 문장 키워드 반영 여부
- 첫 문단 키워드 자연 반복 여부
- 보조 키워드 분산 여부
- 제목/본문 일치 여부
- 검색 의도와 글 목적 일치 여부
- 경험/후기/비교/체크포인트 포함 여부
- 금칙어 없음 확인
- 과장 표현 없음 확인

## 글 목적별 구조

- `정보 전달형`: 공감 도입, 선택 포인트, 확인 기준, 자연스러운 정리 흐름
- `신뢰 형성형`: 불안 해소, 응대 과정, 신뢰 포인트, 문의 부담을 낮추는 마무리
- `상품 홍보형`: 차별점, 핵심 장점, 활용 장면, 부드러운 구매/문의 유도
- `방문 유도형`: 지역/방문 상황, 방문 전 확인 사항, 편한 문의/방문 마무리

## 내부 키워드 최적화

광고주 화면에는 별도 설정으로 노출하지 않지만, 생성 로직 안에는 메인 키워드 최적화 규칙이 들어 있습니다.

- 목표 글자수에 따라 메인 키워드 반복 목표 범위를 다르게 잡습니다.
- 첫 문장에는 메인 키워드를 1회 포함하고, 첫 문단에는 2~3회 자연스럽게 배치합니다.
- 전체 본문에서는 메인 키워드를 5~7회 범위로 분산하고, 같은 표현이 몰리지 않도록 조정합니다.
- 메인 키워드만 반복하지 않고 선택 기준, 활용 포인트, 실전 체크포인트 같은 연관 표현도 함께 사용합니다.
- 생성 결과에는 향후 관리자 화면이나 자동화 브리지에서 쓸 수 있도록 `keywordOptimization` 구조가 함께 남습니다.

## 현재 저장 정책

1차 MVP는 브라우저 `localStorage`에 초안을 저장합니다. 같은 브라우저에서는 새로고침 후에도 보관함에서 초안을 다시 열 수 있지만, 다른 PC나 다른 브라우저와 동기화되지는 않습니다.

콘텐츠 메이커에 저장되는 주요 값은 입력값, 선택한 주제, 선택한 제목, 목표 글자수, 게시용 본문, 해시태그 그룹, 이미지 추천 위치, 내부 전략 메모, 내부 키워드 최적화 정보입니다.

댓글 응답 관리에 저장되는 주요 값은 블로그 URL, 포스팅 제목, 메인 키워드, 사용자 유형, 말투, 금지어, owner nickname, owner aliases, 댓글 목록, 생성된 대댓글, 검토 상태입니다.

## 카테고리별 필수 작성 기준

블로그 올인원은 카테고리별 글의 전문성과 신뢰도를 높이기 위해, 아래 6대 카테고리 생성 시 필수 정보를 확인하고 검증합니다. 사용자가 입력값 또는 작성 프로필에서 해당 필수 정보를 제공하지 않은 경우, AI가 임의로 가상 정보를 생성(지어내기)하지 않고 본문에 `[확인 필요]`로 명시적으로 표기합니다. 또한, 실제 경험이나 방문 여부 등의 정보가 확실하지 않거나 불분명할 경우, 직접 다녀온 것처럼 꾸미지 않고 **'정보 정리형'** 또는 **'방문 전 참고용 안내 글'** 구조로 작성하는 것을 원칙으로 합니다.
