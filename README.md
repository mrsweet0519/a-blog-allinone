# Blog All-in-One 1차 MVP

블로그 운영을 위한 콘텐츠 초안 생성 도구의 실행 가능한 MVP 뼈대입니다. 이번 단계는 실제 AI 완성형 기능보다 화면 구조, 선택형 생성 흐름, 로컬 임시 저장을 빠르게 확인하는 데 초점을 둡니다.

## 실행 방법

Windows PowerShell에서 `npm.ps1` 실행 정책 오류가 나면 `npm` 대신 `npm.cmd`를 사용합니다.

1. 의존성 설치

```bash
npm.cmd run install:all
```

2. 프론트엔드 실행

```bash
npm.cmd run dev:frontend
```

브라우저에서 `http://127.0.0.1:5173`으로 접속합니다.

3. 백엔드 실행

```bash
npm.cmd run dev:backend
```

백엔드 상태 확인 주소는 `http://localhost:4000/api/health`입니다.

## 크몽 시연용 데모 배포

정식 SaaS가 아니라 판매 전 시연용 URL이 목표라면 다음 구조가 가장 단순합니다.

- 프론트엔드: Vercel
- 백엔드: Render Web Service
- 저장 방식: 현재 MVP처럼 브라우저 `localStorage`
- 제외 범위: 회원가입, 결제, DB 저장, 실제 외부 발행 자동화

프론트는 정적 React 앱이므로 Vercel에 잘 맞고, 백엔드는 Express API이므로 Render Web Service로 분리하는 방식이 관리하기 쉽습니다.

### 필요한 환경변수

프론트엔드, Vercel에 입력합니다.

```text
VITE_API_BASE_URL=https://내-render-백엔드주소.onrender.com
```

백엔드, Render에 입력합니다.

```text
CORS_ORIGIN=https://내-vercel-프론트주소.vercel.app
GEMINI_API_KEY=
PEXELS_API_KEY=
```

Render는 실행 포트로 `PORT` 환경변수를 사용할 수 있고, 별도 설정이 없으면 기본 웹 서비스 포트인 `10000`을 사용합니다. 로컬에서 직접 포트를 바꾸고 싶을 때만 `BACKEND_PORT=4000`을 사용합니다.

현재 1차 MVP는 Gemini/Pexels API를 실제 호출하지 않습니다. 나중에 연결할 때도 API 키는 프론트가 아니라 Render 백엔드 환경변수에만 넣어야 합니다.

### 프론트 배포, Vercel

1. Vercel에서 GitHub 저장소 `mrsweet0519/a-blog-allinone`을 Import합니다.
2. Framework Preset은 `Vite`를 선택합니다.
3. Root Directory는 `frontend`로 설정합니다.
4. Build Command는 `npm run build`를 사용합니다.
5. Output Directory는 `dist`를 사용합니다.
6. Environment Variables에 `VITE_API_BASE_URL`을 입력합니다.
7. Deploy 후 Vercel 프로젝트 화면의 Domains 영역에서 접속 URL을 확인합니다.

### 백엔드 배포, Render

1. Render에서 New Web Service를 선택합니다.
2. GitHub 저장소 `mrsweet0519/a-blog-allinone`을 연결합니다.
3. Root Directory는 `backend`로 설정합니다.
4. Build Command는 `npm install`을 사용합니다.
5. Start Command는 `npm start`를 사용합니다.
6. Environment Variables에 `CORS_ORIGIN`을 입력합니다.
7. Deploy 후 Render 서비스 화면의 URL 뒤에 `/api/health`를 붙여 접속합니다.

예:

```text
https://내-render-백엔드주소.onrender.com/api/health
```

`{"ok":true}`가 보이면 백엔드가 정상입니다.

### 로컬 환경변수 예시

예시 파일은 아래에 있습니다.

- `frontend/.env.example`
- `backend/.env.example`

프론트에서 백엔드 API를 사용하려면 `frontend/.env`에 다음 값을 둡니다.

```text
VITE_API_BASE_URL=http://localhost:4000
```

백엔드 CORS 허용 주소를 바꾸려면 Render 또는 로컬 실행 환경에 다음처럼 넣습니다.

```text
CORS_ORIGIN=http://127.0.0.1:5173,http://localhost:5173,https://내-vercel-프론트주소.vercel.app
```

배포 후에는 Vercel URL로 프론트에 접속하고, 설정 화면의 백엔드 API 상태를 눌러 Render 연결 여부를 확인할 수 있습니다.

```text
│  └─ src/styles/       # 전역 Tailwind 스타일
├─ backend/             # Node.js + Express API 뼈대
│  └─ src/
│     ├─ api/           # API 라우터
│     └─ services/      # 콘텐츠 생성 서비스
├─ shared/              # 프론트/백엔드 공용 MVP 설정과 생성 로직
└─ docs/                # 기획/아키텍처/상세 명세 문서
```

## 1차 MVP 범위

- 주제 추천 3개
- 목적별 제목 후보 4개: 정보형, 지역형, 비교형, 클릭형
- 선택한 제목 기준 개요 소제목 4~6개
- 첫 문장 후보 2~3개 선택
- CTA 문장 후보 3개 선택
- 선택한 주제/제목 기준 게시용 최종본 1개
- 게시용 최종본과 내부 전략 메모 분리
- 목표 글자수 선택: 1200자, 1500자, 2000자, 2500자, 직접입력
- 목표 글자수 ±10% 범위 안내
- 내부 관리자용 키워드 최적화 규칙
- 글 목적별 생성 구조: 정보 전달형, 신뢰 형성형, 상품 홍보형, 방문 유도형
- 사용자 유형별 결과 분기: 사업자/매장 홍보, 인플루언서/수익형
- 광고주형 입력값: 브랜드명/매장명, 지역, 핵심 강점, 강조 포인트, CTA 방향
- Pexels 검색 연동을 고려한 이미지 삽입 추천 위치 3개
- 검색패턴 기반 해시태그 그룹
- 기본 대시보드
- 로컬 브라우저 저장소 기반 초안 보관
- 입력 전, 입력 완료, 생성 중, 생성 완료, 수정 중, 저장됨, 복사 완료 상태 표시

## 콘텐츠 메이커 흐름

1. 키워드, 업종/주제, 브랜드명/매장명, 지역, 글 목적, 말투, 핵심 강점, 강조 포인트, CTA 방향, 목표 글자수, 이모지, 금지어를 입력합니다.
2. `주제 생성`을 눌러 추천 주제 3개를 만듭니다.
3. 추천 주제 중 1개를 선택합니다.
4. `제목 생성`을 눌러 정보형, 지역형, 비교형, 클릭형 제목 후보를 만듭니다.
5. 제목 후보 중 1개를 선택합니다.
6. `개요 생성`을 눌러 소제목 4~6개를 만들고, 필요한 소제목을 선택하거나 직접 수정합니다.
7. 첫 문장 후보와 CTA 후보 중 마음에 드는 문장을 선택합니다.
8. `본문 생성`을 눌러 게시용 최종본, 이미지 추천 위치, 검색패턴 해시태그, 내부 운영 메모를 만듭니다.
9. 결과 영역에서 `제목 → 본문 → 해시태그 → 복사` 순서로 확인한 뒤 `복사`를 사용합니다.

## 게시용 최종본과 내부 메모

콘텐츠 메이커 결과는 2단으로 나뉩니다.

- `게시용 최종본`: 광고주가 네이버 블로그에 바로 옮겨 쓸 수 있는 본문입니다. 내부 전략 설명, 메타 문장, 작성자용 안내 문구는 본문에 넣지 않습니다.
- `운영 메모 / 내부 전략 데이터`: 접힌 영역에 보관되는 보조 정보입니다. 글 목적 템플릿, 키워드 반복 범위, 이미지 브리지 슬롯 같은 운영용 데이터를 확인할 수 있습니다.

`복사` 버튼은 게시용 제목, 본문, 해시태그만 복사합니다.

## 이번 품질 튜닝에서 바뀐 점

초보자 기준으로 보면, 이번 변경의 핵심은 "버튼이 더 많아진 것"이 아니라 "복사해서 바로 올릴 수 있는 글에 가까워진 것"입니다.

- 첫 문장은 제목을 반복하지 않고, 메인 키워드가 들어간 독자의 고민 문장으로 시작합니다.
- 첫 문단에는 메인 키워드를 2~3회 자연스럽게 배치하고, 이후 본문에서는 연관 표현을 섞어 과한 반복을 줄입니다.
- 개요 소제목은 `기준 1`, `기준 2`처럼 보이지 않도록 클릭하고 싶은 문장형 제목으로 생성합니다.
- 본문에서는 소제목을 `**소제목**` 형태로 굵게 표시하고, 소제목과 본문 사이에 줄바꿈을 넣어 네이버 블로그에 붙여넣기 편하게 만들었습니다.
- 전체 흐름은 공감, 상황, 기준, 판단, 행동 순서로 이어집니다.
- `사업자/매장 홍보`는 고객 고민, 해결 기준, 신뢰, 방문/문의 CTA가 분명하게 나오고, `인플루언서/수익형`은 경험, 공감, 저장, 비교, 직접 확인 흐름이 더 강하게 나옵니다.
- CTA는 사용자 유형별 3~5개 문장 패턴 중 하나가 적용되어 마무리가 매번 조금 더 자연스럽게 달라집니다.
- 이제 첫 문장과 CTA는 후보를 먼저 보여주고 사용자가 선택한 문장을 최종 본문에 반영합니다.
- 제목은 정보형, 지역형, 비교형, 클릭형으로 구분해 제공합니다.
- 해시태그는 메인 키워드, 지역 조합, 업종/서비스, 검색 의도, 브랜드/매장명, 롱테일 보조 태그로 나누어 생성합니다.
- 내부 메모에는 첫 문장, 첫 문단, 전체 키워드 반복 수, 소제목, 지역명, CTA, 금지어 기준의 최적화 체크가 표시됩니다.
- 이미지 추천에는 `searchKeyword`, `query`, `description`이 함께 들어가 Pexels 같은 이미지 API로 연결하기 쉬운 구조가 되었습니다.

## 네이버 블로그 최적화 구조

이 MVP는 단순 글쓰기 도구가 아니라 네이버 블로그 포스팅 보조 시스템을 목표로 합니다. 그래서 결과물을 제목, 본문, 해시태그, 내부 점검 기준으로 나누어 보여줍니다.

### 제목 최적화 구조

제목 후보는 항상 4가지 역할로 생성됩니다.

- `정보형 제목`: 처음 검색하는 사용자가 기준을 이해할 수 있는 제목
- `지역형 제목`: 지역명과 메인 키워드를 자연스럽게 연결한 제목
- `비교형 제목`: 여러 업체나 선택지를 비교하는 사용자를 위한 제목
- `클릭형 제목`: 숫자, 체크 포인트, 차이처럼 클릭을 유도하는 제목

모든 제목에는 메인 키워드를 포함하되, 키워드만 나열하지 않고 실제 검색 문장처럼 읽히도록 생성합니다.

### 해시태그 그룹 생성 방식

해시태그는 단순 나열이 아니라 검색패턴 기반 그룹으로 생성됩니다.

- `메인 키워드`: 대표 검색어
- `지역 조합`: 지역명 + 키워드, 지역명 + 업종
- `업종/서비스`: 업종, 서비스명, 관련 관리/상담 태그
- `검색 의도`: 방문 전 확인, 상담, 비교, 관리 목적
- `브랜드/매장명`: 상호명 기반 검색 유입
- `롱테일 보조 태그`: 더 구체적인 검색 문장형 태그

화면에서는 그룹별 카드로 보여주고, 복사할 때는 전체 해시태그를 한 번에 복사합니다.

### 본문 최적화 체크 기준

게시용 최종본 아래의 내부 메모에는 최적화 체크가 함께 표시됩니다.

- 첫 문장에 메인 키워드가 1회 포함됐는지
- 첫 문단에 메인 키워드가 2~3회 자연스럽게 들어갔는지
- 전체 본문에서 메인 키워드가 5~7회 범위로 분산됐는지
- 소제목에 메인 키워드가 반영됐는지
- 지역명이 본문이나 검색 구조에 반영됐는지
- 선택한 CTA가 본문에 들어갔는지
- 금지어가 포함되지 않았는지

현재 검색패턴 기반 태그와 최적화 체크는 1차 MVP 기준의 규칙 기반 로직입니다. 향후에는 네이버 데이터랩, 검색광고 API, 실제 검색량 데이터와 연결해 키워드 조합과 태그 추천 정확도를 높일 수 있습니다.

## 글 목적별 구조

- `정보 전달형`: 공감 도입, 선택 포인트, 확인 기준, 자연스러운 정리 흐름
- `신뢰 형성형`: 불안 해소, 응대 과정, 신뢰 포인트, 문의 부담을 낮추는 마무리
- `상품 홍보형`: 차별점, 핵심 장점, 활용 장면, 부드러운 구매/문의 유도
- `방문 유도형`: 지역/방문 상황, 방문 전 확인 사항, 편한 문의/방문 마무리

## 내부 키워드 최적화

광고주 화면에는 별도 설정으로 노출하지 않지만, 생성 로직 안에는 메인 키워드 최적화 규칙이 들어 있습니다.

- 목표 글자수에 따라 메인 키워드 반복 목표 범위를 다르게 잡습니다.
- 첫 문장에는 메인 키워드를 1회 포함하고, 첫 문단에는 2~3회 자연스럽게 배치합니다.
- 전체 본문에서는 메인 키워드를 5~7회 범위로 분산하고, 같은 표현이 몰리지 않도록 조정합니다.
- 메인 키워드만 반복하지 않고 선택 기준, 활용 포인트, 실전 체크포인트 같은 연관 표현도 함께 사용합니다.
- 생성 결과에는 향후 관리자 화면이나 자동화 브리지에서 쓸 수 있도록 `keywordOptimization` 구조가 함께 남습니다.

## 현재 저장 정책

1차 MVP는 브라우저 `localStorage`에 초안을 저장합니다. 같은 브라우저에서는 새로고침 후에도 보관함에서 초안을 다시 열 수 있지만, 다른 PC나 다른 브라우저와 동기화되지는 않습니다.

저장되는 주요 값은 입력값, 선택한 주제, 선택한 제목, 목표 글자수, 게시용 본문, 해시태그 그룹, 이미지 추천 위치, 내부 전략 메모, 내부 키워드 최적화 정보입니다.
